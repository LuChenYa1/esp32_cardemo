## 关键技术决策

### 1. 硬件平台选择：ESP32-S3

**决策**: 从STM32迁移到ESP32-S3

**理由**:
- 更强的处理能力（双核240MHz vs 单核72MHz）
- 内置WiFi和蓝牙功能，便于扩展
- 丰富的外设接口（ADC、PWM、UART、I2C、SPI）
- 成熟的ESP-IDF开发框架和FreeRTOS支持
- 更低的成本和更好的生态系统

**影响**:
- 需要重新设计定时器系统（STM32 TIM → ESP32硬件定时器）
- 需要适配ADC接口（STM32 ADC → ESP32 ADC2）
- 需要处理GPIO限制（ADC2与WiFi冲突、GPIO6-11不可用）
- 获得了更好的任务调度能力（FreeRTOS）

### 2. 实时任务调度架构：硬件定时器 + FreeRTOS

**决策**: 使用ESP32硬件定时器实现高频实时任务，FreeRTOS管理低频外设任务

**理由**:
- 巡线控制需要严格的实时性（1ms灰度扫描，10ms PD控制）
- FreeRTOS任务调度有不确定性，不适合高频实时任务
- 硬件定时器中断优先级高，时序精度好
- 低频外设任务（数码管、舵机）可以使用FreeRTOS任务

**架构**:
```
硬件定时器中断层（最高优先级）
├── Timer 0 (3ms): 灰度扫描、转弯检测
└── Timer 1 (10ms): PD控制、转弯状态机

FreeRTOS任务层（低优先级）
├── ADC采样任务 (1ms): 阻塞式ADC读取
├── 语音模块任务: 接收语音命令
├── 数码管显示任务 (5s轮播): 显示系统状态
├── 舵机控制任务 (3s往复): 控制舵机动作
└── 交通灯控制任务 (3s切换): 切换交通灯状态
```

**优势**:
- 保证了巡线控制的实时性和稳定性
- 充分利用了FreeRTOS的任务管理能力
- 清晰的优先级分层，避免任务冲突

### 3. ADC读取策略：阻塞式读取 + 共享变量缓存

**决策**: ADC采样任务阻塞式读取，中断中读取缓存值

**理由**:
- ESP32 ADC读取需要时间（约500微秒），不能在中断中阻塞
- 中断中直接调用ADC会导致中断延迟，影响实时性
- 使用共享变量缓存ADC值，中断中快速读取

**实现**:
```c
// ADC采样任务（FreeRTOS任务，1ms周期）
void adc_sampling_task(void *pvParameters) {
    while (1) {
        // 阻塞式读取ADC
        adc_oneshot_read(adc2_handle, ADC_CHANNEL_9, &left_raw);
        adc_oneshot_read(adc2_handle, ADC_CHANNEL_7, &right_raw);
        
        // 原子写入共享变量
        atomic_store(&g_adc_left_cached, left_raw);
        atomic_store(&g_adc_right_cached, right_raw);
        
        vTaskDelay(pdMS_TO_TICKS(1));
    }
}

// Timer 0中断（1ms周期）
void IRAM_ATTR timer0_isr_handler(void *arg) {
    // 原子读取缓存值（快速，无阻塞）
    uint16_t left = atomic_load(&g_adc_left_cached);
    uint16_t right = atomic_load(&g_adc_right_cached);
    
    // 执行转弯检测...
}
```

**优势**:
- 中断执行时间短，不影响实时性
- ADC读取失败不会阻塞中断
- 数据传递清晰，易于调试

### 4. 线程安全策略：原子变量保护

**决策**: 使用C11原子变量（_Atomic）保护跨中断/任务的共享数据

**理由**:
- 原子操作硬件支持，性能好
- 避免使用互斥锁（中断中不能使用）
- 代码简洁，易于理解

**保护的共享变量**:
```c
static _Atomic uint16_t g_adc_left_cached = 4095;
static _Atomic uint16_t g_adc_right_cached = 4095;
static _Atomic uint8_t g_turn_type = 0;
static _Atomic bool g_turning_in_progress = false;
static _Atomic TurnState_t g_turn_state = TURN_IDLE;
```

**优势**:
- 保证了数据一致性，避免竞态条件
- 性能开销小，适合实时系统
- 符合现代C语言标准

### 5. GPIO管理机制：集中管理 + 冲突检测

**决策**: 实现GPIO管理器，集中管理引脚分配和冲突检测

**理由**:
- ESP32-S3可用GPIO有限，需要合理分配
- 多个组件可能尝试使用同一GPIO，导致冲突
- 集中管理便于调试和维护

**实现**:
```c
// 组件初始化时注册GPIO
esp_err_t my_component_init(void) {
    esp_err_t ret = gpio_manager_register(
        GPIO_NUM_18,
        GPIO_FUNC_ADC,
        "gray_sensor",
        "左灰度传感器"
    );
    
    if (ret != ESP_OK) {
        ESP_LOGE(TAG, "GPIO18已被占用");
        return ESP_FAIL;
    }
    
    // 继续初始化...
}
```

**优势**:
- 启动时自动检测GPIO冲突，避免运行时错误
- 提供GPIO分配表，便于调试和文档维护
- 强制组件注册GPIO，规范开发流程

### 6. 错误处理策略：分级处理 + 安全模式

**决策**: 根据错误严重程度分级处理，关键错误进入安全模式

**错误分级**:
- **低级错误**: 保持上次值，记录错误（如ADC读取失败）
- **中级错误**: 重试机制，记录错误（如舵机通信超时）
- **高级错误**: 停止电机，进入安全模式（如ADC连续失败100次）
- **致命错误**: 记录错误，停止启动（如定时器初始化失败）

**安全模式**:
```c
void enter_safe_mode(void) {
    // 停止所有电机
    set_motor_speed(0, 0);
    set_motor2_speed(0, 0);
    set_motor3_speed(0, 0);
    set_motor4_speed(0, 0);
    
    ESP_LOGE(TAG, "进入安全模式：ADC连续失败");
    
    // 进入死循环，等待重启
    while (1) {
        vTaskDelay(pdMS_TO_TICKS(1000));
    }
}
```

**优势**:
- 提高了系统鲁棒性，避免失控
- 分级处理避免过度反应
- 安全模式保护硬件和人员安全

## 已知问题和限制

### 硬件限制

#### 1. ADC2与WiFi冲突（关键）

**问题**: ADC2通道在WiFi启动后无法使用

**影响**: 当前灰度传感器使用GPIO18/20（ADC2通道），无法同时使用WiFi

**解决方案**:
- 短期：不使用WiFi功能
- 长期：将灰度传感器改用GPIO47/48（ADC1通道）

**状态**: 已记录，待改进

#### 2. GPIO6-11不可用（关键）

**问题**: GPIO6-11连接到SPI Flash，不能用于应用程序

**影响**: 可用GPIO数量减少

**解决方案**: 使用其他GPIO或通过I2C扩展（PCF8574）

**状态**: 已规避

#### 3. GPIO34-39仅输入（重要）

**问题**: GPIO34-39只能配置为输入模式

**影响**: 不能用于PWM输出、数字输出

**解决方案**: 仅用于ADC采样、数字输入

**状态**: 已规避

### 软件限制

#### 1. 中断中不能使用阻塞API（关键）

**问题**: 中断服务函数中不能调用vTaskDelay、printf、mutex等阻塞API

**影响**: 中断中不能直接调用某些ESP-IDF API

**解决方案**: 
- 使用IRAM_ATTR属性标记中断函数
- 避免在中断中调用阻塞API
- 使用共享变量传递数据到任务中处理

**状态**: 已规避

#### 2. FreeRTOS任务栈大小限制（重要）

**问题**: 任务栈大小不足会导致栈溢出

**影响**: 系统崩溃或重启

**解决方案**:
- 合理设置任务栈大小（推荐4096字节以上）
- 启用任务栈溢出检测（Task Watchdog Timer）
- 避免在任务中使用大数组

**状态**: 已配置

#### 3. UART0与RS485互斥（一般）

**问题**: UART0同时用于日志打印和RS485通信

**影响**: 舵机控制时需要关闭日志输出

**解决方案**:
- 舵机任务中关闭ESP_LOG输出
- 或使用UART1作为日志输出

**状态**: 已规避

### 已知Bug

目前无已知的严重Bug。
